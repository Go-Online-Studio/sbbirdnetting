2. REQUIRED AUDIT PROCESS
=====================================================================

First create a concise audit map:
- Actual stack and framework.
- Metadata implementation.
- Schema implementation.
- Sitemap and robots implementation.
- Redirect and hosting configuration.
- Asset strategy.
- JavaScript/CSS bundle strategy.
- Third-party scripts and dependencies.
- Known build/test commands.

Then classify every finding:
- P0: blocks launch, indexing, security, or core functionality.
- P1: serious SEO/GEO/performance/accessibility/security issue.
- P2: important improvement.
- P3: polish or optional optimization.

For each issue, provide:
- File path and exact location when available.
- What is wrong.
- Why it matters.
- Exact fix applied or exact replacement code.
- Verification performed.

=====================================================================
3. PRODUCTION ENVIRONMENT AND DEPLOYMENT CLEANUP
=====================================================================

Audit and fix:
- localhost, staging, preview, GitHub Pages, temporary, old-domain, or repository-subfolder URLs.
- Incorrect asset paths after production deployment.
- Broken internal links.
- Duplicate URL versions.
- Missing or wrong canonical URLs.
- Environment variable exposure.
- Source maps or debug artifacts that expose sensitive information.
- Console logs, test routes, demo pages, placeholder content, and development-only comments where inappropriate.
- Deployment configs that do not match the actual host.

Redirect requirements:
- Force HTTPS.
- Enforce preferred www/non-www canonical domain.
- Normalize trailing slash behavior consistently.
- Redirect old URLs to current canonical URLs when known.
- Prevent duplicate versions of the same page.

Generate or fix the correct config when applicable:
- Apache / LiteSpeed: .htaccess
- Nginx: nginx.conf
- Vercel: vercel.json or framework redirects
- Netlify: netlify.toml or _redirects
- Cloudflare Pages: _redirects or Cloudflare rules
- Next.js: next.config.js redirects when appropriate

=====================================================================
4. TECHNICAL SEO AUDIT AND FIXES
=====================================================================

For every indexable page or route, verify and fix:
- Exactly one meaningful H1.
- Logical heading hierarchy.
- Unique title tag.
- Unique meta description.
- Self-referencing absolute canonical URL.
- Correct robots meta tag or X-Robots-Tag.
- Open Graph title, description, URL, image, type, locale, and site name.
- Twitter/X summary_large_image metadata.
- Descriptive URL slug.
- Semantic HTML5 structure.
- Breadcrumbs where hierarchy exists.
- Crawlable internal links.
- No empty headings.
- No important text only inside images.
- No placeholder or Lorem ipsum content.
- No duplicate title/description/canonical mistakes.
- No orphan important pages.
- 404 page exists and returns correct status where applicable.

Canonical rules:
- Use HTTPS.
- Use the preferred domain variant.
- Match the live route exactly.
- Do not canonical all pages to the homepage.
- Do not point service/product/article pages to the wrong URL.
- Use canonical consolidation only when intentionally merging duplicate pages.

Images and social sharing:
- OG and Twitter images must be absolute URLs.
- Social images should have stable dimensions and be crawlable.
- Logo and favicon should be configured correctly.

=====================================================================
5. SITEMAP AND ROBOTS.TXT
=====================================================================

Create or fix sitemap.xml or framework sitemap generation.

Sitemap requirements:
- Include all public indexable URLs.
- Include service, product, category, article, blog, location, and detail pages when public.
- Use absolute canonical URLs.
- Include lastmod where trustworthy.
- Exclude admin, staging, test, private, duplicate, thank-you, cart, checkout, preview, and filtered/search URLs unless intentionally indexable.
- Auto-update from route/config/CMS data when the stack supports it.
- Validate that sitemap URLs return 200 and are canonical.

Robots.txt requirements:

User-agent: *
Allow: /

Sitemap: [CANONICAL_DOMAIN]/sitemap.xml

Also:
- Disallow private, admin, staging, preview, test, internal API, and generated duplicate paths where relevant.
- Do not block CSS, JS, images, or resources required for rendering.
- Do not use robots.txt to hide private or sensitive data.
- If blocking a page from search is required, use noindex or authentication instead of only robots.txt.
- If AI crawler policy is requested, document it clearly and explain limits. Robots rules are not security controls and may not be followed by all crawlers.

- Sitemap and robots allow discovery of important pages.

9. CORE WEB VITALS AND PERFORMANCE AUDIT
=====================================================================

Measure when tools are available. Otherwise inspect code and assets carefully.

Targets:
- LCP: 2.5s or faster where realistically possible.
- INP: 200ms or faster where realistically possible.
- CLS: 0.1 or lower.
- Lighthouse: aim for 90+ in Performance, Accessibility, Best Practices, and SEO where realistic.

LCP:
- Identify the actual LCP element.
- Optimize or replace oversized hero images.
- Use preload or priority only for the true LCP asset.
- Avoid hero sliders/video backgrounds as the first meaningful content.
- Reduce render-blocking CSS and JavaScript.
- Use server rendering or static output for above-fold content when SEO matters.

CLS:
- Add width, height, aspect-ratio, or reserved boxes for images, iframes, embeds, ads, banners, maps, forms, review widgets, and sliders.
- Ensure fonts use font-display: swap.
- Avoid injecting content above existing content after load.

INP and JavaScript:
- Remove unused JavaScript.
- Split or defer non-critical scripts.
- Avoid duplicate libraries.
- Replace heavy libraries with CSS or lightweight code when possible.
- Throttle/debounce expensive handlers.
- Use passive listeners for scroll/touch where appropriate.
- Audit chat widgets, maps, analytics, sliders, animation libraries, and embeds.

CSS and fonts:
- Remove unused CSS.
- Minify CSS.
- Avoid large icon font bundles.
- Limit font families, weights, and blocking font requests.

Images/media:
- Convert oversized images to WebP/AVIF when suitable.
- Use responsive srcset/sizes.
- Lazy-load below-fold images and iframes.
- Use decoding="async" where appropriate.

Caching:
- Add correct cache-control for static assets.
- Use hashed filenames or versioning for long-lived assets.
- Use compression: Brotli or gzip.
- Add CDN configuration if relevant.
- Use a service worker only when it is appropriate and correctly versioned.

10. ACCESSIBILITY AUDIT
=====================================================================

Verify and fix:
- Keyboard navigation.
- Visible focus states.
- Skip-to-content link where useful.
- One H1 per page and logical headings.
- Semantic buttons and links.
- Accessible nav, menus, dropdowns, modals, carousels, and forms.
- Labels, error messages, autocomplete, and validation on forms.
- Color contrast at WCAG AA where possible.
- Alt text for meaningful images and alt="" for decorative images.
- ARIA only when needed and not contradictory.
- Reduced motion support.
- Touch targets around 44px where practical.
- No overlapping text or controls at mobile/tablet/desktop/wide viewports.

=====================================================================
11. SECURITY AND VULNERABILITY AUDIT
=====================================================================

Use OWASP-style thinking and the actual stack.

Dependency and supply chain:
- Inspect package.json, lockfiles, CDN scripts, plugins, themes, and build tools.
- Run npm audit, pnpm audit, yarn audit, composer audit, pip audit, or the relevant package audit when available.
- Remove unused dependencies.
- Avoid abandoned or suspicious packages.
- Pin versions where appropriate.
- Use Subresource Integrity for CDN scripts/styles when feasible.

Secrets and sensitive data:
- Search for API keys, tokens, passwords, private URLs, emails not meant for public use, .env leaks, credentials, private keys, and debug dumps.
- Ensure secrets are server-side environment variables, not client-side code.
- Ensure public config is intentionally public.

XSS and injection:
- Escape user-generated content.
- Avoid unsafe innerHTML/dangerouslySetInnerHTML unless sanitized.
- Sanitize CMS HTML.
- Validate form inputs on client and server where a backend exists.
- Avoid open redirects and untrusted URL injection.

Forms, APIs, auth, and data:
- Check CSRF protection where relevant.
- Check authentication and authorization boundaries.
- Check rate limiting or spam protection for public forms/APIs.
- Check CORS policy.
- Check file upload restrictions if uploads exist.
- Check payment and checkout flows if present.
- Do not collect unnecessary personal data.

Headers and browser protections:
- Add or recommend Content-Security-Policy.
- Add Strict-Transport-Security after HTTPS is confirmed.
- Add X-Content-Type-Options: nosniff.
- Add Referrer-Policy.
- Add Permissions-Policy.
- Add frame-ancestors in CSP or X-Frame-Options when appropriate.
- Avoid overly permissive CORS.

Third-party scripts:
- Audit analytics, chat, maps, reviews, ads, widgets, and pixels.
- Defer or delay non-critical third-party scripts.
- Confirm privacy/cookie consent requirements if applicable.
- Remove scripts that are unused, slow, insecure, or not approved.

Static site notes:
- Even static sites can leak secrets, expose old files, use vulnerable JS libraries, or misconfigure headers.
